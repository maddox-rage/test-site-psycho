свойства каналов:
- goroutine-safe(потоковая безопасность), с каналами могут одно временно работать несколько го-рутин, благодаря этому можно не заморачиваться с примитивами синхронизации(локами и мьютексами)(buf, reciveqx, sendqx, mutex) - достигли с помощью hchan mutex
- хранение элементов, которые хранятся в семантике FIFO(first int first out) - достигли с помощью кольцевой очереди (hchan buf)
- передача данных между го-рутинами - достигли с помощью sendDirect() и операций с буфером 
- блокировка го-рутин - достигли с помощью очередей sendq/recvq, структуры sudog и обращений к планировщику(gopark() and goready())

поскольку с каналами могут работать несколько го-рутин, поэтому мы храним всё в куче(heap)
параметры hchan:
- qcount - кол-во элементов, которые находятся в канале
- dataqsize - размер канала, если канал буферезирован, то это количество элементов 
- buf - буфер, в котором храним канал, если канал буферезирован 
- closed - только два значения, если 1 - закрыт, если 0 - открыт(тк работаем с атомиками, этот параметр uint32, а не булл, тк атомики с булл не работают)
- reciveq and sendq(тип данных waitq(поля: указатели на первый и последний элементы связанного списка)) - очередь го-рутин
- recuveqx and sendqx - представляют из себя счетчики на элемент буфера, в который будет производиться след запись или чтение
- mutex - когда что-то делаем с каналом, блокируем его, освобождаем - разблокируем
канал представляет из себя указатель на структура hchan(позволяет работать с каналом нескольким го-рутинам)
для реализации семантики fifo нужна очередь
буфер представляет из себя кольцевую очередь, recvx and sendx может представить как указатели на элементы очереди
как только что-то записываем в канал, sendx увеличивается на единицу, после чего запись и снова увеличение sendx на единицу. Когда буфер заканчивает происходит обнуление
если нужно считать данные, то мы вычитываем с той ячейку, на которую указывает recvx и так же увеличивается на единицу, выгоднее бегать по кругу, чем постоянно начинать с начала
как происходит запись с го-рутин в канал:
1) происходит блокировка mutex
2) копируем данные в канал(именно копия, если это не ссылочный тип данных)
3) разблокировка mutex
так же с получением данных
го-рутины не могут работать параллельно, тока системные треды и то с оговоркой 
у каждого треда будет своя очередь го-рутин, которая называется runqueue. контекст переключается между го-рутинами, которые находятся в очереди и они выполняются по очереди, всем этим руководит планировщик(scheduler)
когда данные некуда класть, го-рутина ставится на паузу
чтобы поставить го-рутину на паузу, вызывается функция gopark(), которая обращается к планировщику, он меняет состояние рутины с running on waiting и дальше он разрывает связь между этой рутиной и системным тредом, после чего отдаёт этот тред следующей рутине, которая стоит в очереди 
мы добавляем рутины в очередь sendq
элементы связанного списка являются структуры sudog(параметры g and elem), g - спящая рутина, elem - указатель на ячейку памяти, в которую мы поместили временно данные, чтоб их считала другая рутина
если бы нам нужно было бы считать данные, то в elem записывается ячейка куда нам нужно положить данные 
когда рутина ушла на паузу и пришла другая, которой нужно считать данные, то всю работу спящей проделывает рутина, которая пришла за данными, после чего спящая уходит дальше
сначала считываются данные, которые не поместились в кольцевую очередь
для пробуждения спящей рутины используется функция goready(), которая меняет состояние на ranneable через планировщика, после чего рутина попадает в очередь(runqueue)
когда нужно считать данные, а буфер пустой, то reader становится на паузу и помещается в recvq с помощью gopark() так же формируется структура sudog. когда данные приходят, то они не кладутся в буфет, а сразу летят в стек рутины, которой эти данные нужны, это нужно для оптимизации, это единственное место где одна рутина получает доступ к стеку другой рутины 
всё выше для буферезированных каналов 
для unbuffered каналов:
нет буфера и данные напрямую летят из стека одной рутины в стек другой рутины с помощью функции sendDirect() и не важно кто пришел первый, не зависит от этого

select: 
все кейсы сортируются случайным образом 
чтение происходит в неблокирующем режиме, за чтение из канала отвечает функция chanrecv(hchan, poiter, false), третий параметр blocking, если true то чтение в блокирующем режиме. Если тру, то когда не получилось получить данные с канала, то рутина встаёт на паузу, если false то функция так же возвращает false, нужно для того, чтобы обойти все кейсы и проверить откуда можем считать данные или записать их
если есть параметр дефолт, то вся конструкция select не блокирующая, не происходит случая, когда рутина встает на паузу

закрытие канала:
проверяем инициализирован ли канал, если нет, то паника
блокируем mutex
проверяем не закрыт ли канал, если да то паника
меняем флаг set.ch.closed =true
отпускаем всех ридеров
отпускаем всех врайтеров и они паникуют
разблокируем mutex и все спящие рутины 
функция, которая за это отвечает closechan()
