методы отличаются от обычной функции специальным параметром - получателем. в определении метода получатель указывается сразу после ключевого слова func 
Обычно используют указатель: так Go не приходится копировать всю структуру, а метод может ее изменить.
Выбор между использованием значения или указателя в методах может быть сложным, особенно для начинающих программистов на Go. Если сомневаетесь, используйте указатель, но бывают случаи, когда использование значения имеет смысл, обычно по соображениям эффективности, например, для небольших неизменяемых структур или значений базового типа. Несколько полезных рекомендаций:
- Если в качестве получателя используется map, func или chan, не используйте указатель на них. Если в качестве получателя используется срез, и метод не перераспределяет память для этого среза, не используйте указатель на него.
- Если метод требует изменения получателя, то получателем должен быть указатель.
- Если получателем является структура, содержащая sync.Mutex или аналогичное поле синхронизации, то для предотвращения копирования получателем должен быть указатель.
- Если получателем является большая структура или массив, то указательный получатель будет более эффективен. Насколько большим считается большой массив? Предположим, что это эквивалентно передаче всех его элементов в качестве аргументов методу. Если это кажется слишком большим, значит, это слишком большой и для получателя.
- Может ли функция или методы, вызываемые одновременно или при вызове из этого метода, изменять получателя? Тип значения создает копию получателя при вызове метода, поэтому внешние обновления не будут применяться к этому получателю. Если изменения должны быть видны в исходном получателе, получатель должен быть указателем.
- Если получателем является структура, массив или срез, и какой-либо из его элементов представляет собой указатель на что-то, что может изменяться, предпочтительнее использовать указатель в качестве получателя, поскольку это сделает намерение более понятным для читателя.
- Если получателем является небольшой массив или структура, которая по своей природе является типом значения (например, тип time.Time), без изменяемых полей и указателей, или просто простой базовый тип, такой как int или string, то использование получателя значения имеет смысл. Получатель значения может уменьшить количество генерируемого мусора; если значение передается методу значения, можно использовать его копию в стеке вместо выделения памяти в куче. (Компилятор пытается разумно избегать этого выделения, но ему это не всегда удается.) По этой причине не следует выбирать тип получателя значения без предварительного профилирования.
- Не смешивайте типы получателей. Для всех доступных методов выбирайте либо указатели, либо структуры.
- Наконец, если сомневаетесь, используйте указательный приемник.

В Go нет наследования. Вместо него активно используется композиция — когда новое поведение собирают из кирпичиков существующего.
По-хорошему, **usage** _расширяет_ **counter** — отношение между ними больше похоже на наследование, чем на композицию. В Go в таких случаях используют _встраивание_(embedding)
Здесь функция **inc** работает как замыкание — она имеет доступ к внутренним полям **c**, хотя снаружи они не видны.
Такая штука называется _метод-значение_ (method value). Используется редко. Но может пригодиться, чтобы разрешить клиенту вызывать метод структуры, не давая при этом доступ к ее полям.
Здесь функция **inc** принимает первым аргументом получателя метода — значение **x** типа ***counter** — и дальше работает как метод, увеличивая значение **x.value**.

Такая штука называется _метод-выражение_ (method expression). На практике встречается еще реже, чем метод-значение.
