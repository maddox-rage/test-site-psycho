rest(representational state trasfer) - передача репрезентативного состояния - **архитектурный**стиль для взаимодействия частей приложения по сети, описанный Роем Филдингом, одним из авторов HTTP.
REST API — это API, которое следует набору правил (принципов), благодаря чему становится предсказуемым и единообразным для клиентов.
## Ключевые принципы REST

1. Клиент‑серверная модель
    - Клиент инициирует запросы (браузер, приложение и т.п.), сервер принимает запрос, обрабатывает и возвращает ресурс (HTML, JSON, картинка и т.д.).
    - Клиент и сервер независимы: сервер можно переписать, не ломая клиента, если формат запросов/ответов прежний.
2. Отсутствие состояния (Stateless)
    - Сервер не хранит контекст между запросами, каждый запрос содержит всю нужную информацию (пример с переходом по страницам каталога ?page=10, ?page=9).
    - Это разгружает сервер, но часть информации можно держать у клиента или промежуточных узлов.
3. Кэширование (Caching)
    - Сервер может сказать клиенту (браузеру) сохранить, например, картинку на какое‑то время, чтобы не запрашивать её каждый раз.
4. Многоуровневая система (Layered System)
    - Между клиентом и «главным» сервером могут быть прокси, балансировщики, другие сервера; каждый видит только ближайших соседей, а клиент не знает, до какого именно сервера дошел запрос.
5. Единство интерфейса (Uniform Interface)  
    Состоит из нескольких аспектов:
    - Каждый ресурс имеет уникальный URL (идентификатор).
    - В запросе явно указывается, что нужно сделать и какие данные нужны; при создании/редактировании сущности отправляется полное представление ресурса (например, весь пост блога).
    - Сообщения самодостаточны: ответ содержит все, чтобы клиент понял, как его обрабатывать (например, тип контента).
    - В ответе можно передавать ссылки на связанные ресурсы через гиперссылки.
## HTTP и REST на примере интернет‑магазина

- Клиент (браузер) отправляет HTTP‑запросы на endpoint’ы сервера: например, POST на merion-shop.ru/order при оформлении заказа
- В запросе есть: URL, метод (POST, GET, PUT, DELETE), заголовки (служебные данные, авторизация), тело (обычно JSON с данными заказа, адресом и т.п.).
- В ответе: статус‑код, заголовки, тело (например, id созданного заказа)
## Статус‑коды (основные группы)
- 1xx — информационные (пример: 101 при установке WebSocket).
- 2xx — успех: 200 OK, 201 Created (для успешно созданного ресурса), 204 No Content (успех без тела ответа).
- 3xx — редиректы: 307 временный, 308 постоянный.
- 4xx — ошибка клиента: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests, шуточный 418 I’m a teapot.
- 5xx — ошибка сервера: 500 Internal Server Error, 503 Service Unavailable, 504 Gateway Timeout, 502/504 проблемы с gateway.
## Методы и работа с ресурсами
- POST — создание ресурса (создаем заказ, сервер возвращает 201 и id).
- GET — получение данных: например, посмотреть корзину; параметры передаются в URL, а не в тело.
- PUT — полное обновление ресурса: отправляем все поля заказа по URL вида /order/12345, получаем 200 или 204.
- DELETE — удаление ресурса по тому же URL /order/12345 без тела запроса

## Параметры пути и строки запроса

- Path parameters: часть URL, описывающая ресурс, например /catalog/{category_id} → /catalog/12, сервер берет id и возвращает товары этой категории.
    
- Query parameters: фильтры и опции в конце URL после ?, в виде ключ=значение, разделенные & — пример: /search?query=iphone&limit=10.
## Итоговая мысль автора
- REST — это набор договоренностей и рекомендаций: можно технически нарушать (делать GET для создания, возвращать «капибару» с 500), но тогда это уже не REST и возникают проблемы предсказуемости
- REST полезен и распространен, но не всегда целесообразен; автор предлагает задуматься, где он не подходит, и в шутливой форме призывает «отправлять POST на лайк и PUT на подписку».

url - uniform resource locator 
api (application programming interface) - код, который позволяет двум приложениям обмениваться данными с сервера(программный интерфейс приложения )
http (hypertext transfer protocol) - протокол передачи гипертекста, протокол прикладного уровня для обмена данными между клиентом и сервером  
json(js object notation) - текстовый формат обмена данными, основанный на синтаксисе js. был разработан в начале 2000х Дугласом Крокфордом 
sql (structured query lanuage) - язык структурированных запросов 
handler - обработчик запросов к службе, вызывается, когда службе приходит запрос. он выполняет необходимые действия в соответствии с запросом, после чего сообщает новое состояние 
middleware (промежуточное программное обеспечение, связующее программное обеспечение) - слой или комплекс технологического программного обеспечения для обеспечения взаимодействия между различными приложениями, системами или компонентами 
package в go - основной механизм организации кода, который позволяет структурировать программу, разделяя её на независимые логические блоки. пакеты позволяют разделять код на модули и управлять зависимостями между ними 
struct - сложный тип данных, который позволяет объединить несколько переменных разных типов под одним именем.
connection pool - кэш соединения с бд, которые можно использовать повторно, вместо того, чтобы каждый раз создавать новое соединение, приложение берет его из пула, выполняет операции с бд, а затем возвращает соединение обратно в пул для будущего использования 
repository pattern - выступает прослойкой между логикой приложения и источником данных. класс, через который код получает доступ к данным, не зная деталей их хранения
