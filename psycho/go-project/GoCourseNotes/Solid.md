#S
single responsibility principle(принцип единственной ответственности) гласит, что класс или модуль должен иметь только одну причину для изменения. каждый класс или функция должны решать только одну задачу. когда компонент выполняет тока одну задачу, то его проще изменять, не затрагиваю другой функционал.

#O
open/closed principle(принцип открытости закрытости) - программные сущности(классы, модули, функции) должны быть открыты для расширения, но закрыты для изменения. нужно, чтобы для добавления новой функциональности не требовалось менять существующий код.

#L
liskov substitution princple (принцип подстановки) - объекты должны быть заменяемы на экземпляры их подтипов без изменения правильности работы программы. любой код, который использует родительский класс, должен так же хорошо работать с объектами дочернего класса - объекты базового класса должны заменяться объектами производных классов без изменения функциональности программы

#I
interface segregation principle(принцип разделения интерфейсов) - пользователи не должны зависеть от интерфейсов, которые они не используют. вместо одного наполненного лучше иметь несколько тонких и специализированных 

#D
Dependency inversion principle(принцип инверсии зависимостей) высокоуровневые модули не должны зависеть от низкоуровневых, оба типа модулей должны зависеть от абстракций.
