#Эвакуация
при росте мапы элементы не переносятся сразу, а постепенно по мере работы с мапой. создаётся массив в два раза больше, в hmap хранятся ссылки на новый и старые бакеты и флаг growing. как только все элементы будут перенесены, то ссылка на старый массив удаляется и режим роста заканчивается.

#устройства_бакет_и_сложность_операций 
каждый бакет хранит в себе 8 пар ключей и значений, служебные байты tophash и указатель на overflow бакет, если элементов больше 8
*вставка* 
- вычисляется хэш ключа, по старшим битам выбирается бакет.
- внутри бакета линейный просмотр до 8 слотов, при необходимости переход на overflow бакет 
- сложность близка к константе, но при длинных цепочках латентность растёт
*чтение*
- хэш - бакет - линейный поиск
- сложность константа, но в худшем случаи деградирует до 0(k) - где k - длинна списка
*удаление*
- находится лот, помечается как пустой, счётчик обновляется, по необходимости запускается рост\сжатие
**Tophash** — это служебный массив из 8 байтов (`uint8[8]`) в каждом бакете map (до Go 1.24), который хранит **верхние 8 бит хеша** каждого ключа для быстрого отсева неподходящих слотов без сравнения полных ключей.
Зачем нужен tophash
**Главная идея**: сравнение строк/структур — дорогое. Tophash позволяет **отсечь 99.6% кандидатов** за 1 байт на слот вместо полного `==` ключей.

#структура_Hmap
не хранит ключи и значения, только метаданные
*ключевые поля*
- count - число элементов
- B - логарифм числа бакетов 2^B
- buckets - указатель на массив бакетов
- oldbuckets - указатель на старый массив при росте
- nevacuate - индекс следующего бакета, который нужно эвакуировать
- hash0 - сид для хэш-функции

#определение_каналов_и_виды_каналов
*виды*:
- небуферизованный (make(chan T))
	- daraqsiz = 0, buf = nil
	- отправка и приём всегда синхронны
- буферизованный (make(chan T, n))
	- есть буфер n, отправка блокируется только при полном буфере, чтение - при пустом 
выделяют канала по направлению: только чтение или запись
закрытый открытый

#структура_hchan
*поля*:
- qcount - сколько элементов сейчас в буфере
- dataqsiz - размер буфера( в небуферизованных - 0)
- buf - указатель на кольцевой буфер(nil в небуферизованных)
- elemsiz, elemtype - размер и тип элемента
- sendx, recvx - индексы записи и чтения в кольцевом буфере
- sendq, recvq - очереди горутин, ожидающих отправка или приём(списки sudog)(работают через прямое спаривание горутин из sendq/recvq)
- closed - флаг закрытия канала
- mutex -защита от одновременного доступа 

#для_чего_нужен_select
select позволяет ждать сразу несколько операций с каналами и реагировать на ту, которая станет готовая первой
*возможности:*
- мультиплексирование несколько входов: читаем из того канала, где данные пришли раньше
- реализация таймаутов
- реагирование на сигнал завершения 
- неблокирующие операции: default-ветка

#кольцевой_буфер_в_канале
буферизированный канал хранит элементы в кольцевом массиве, на который указывает buf 
как работает:
- sendx индекс, куда писать элементы, recvx откуда читать
- при отправке значение кладется по индексу sendx, затем sendx = (sendx+1)%dataqsiz, qcount++
- при чтении берется элемент по индексу recvx, recvx = (recvx+1)%dataqsiz, qcount--
- когда они достигают конца массива, они возвращаются обратно
такой буфер позволяет эффективно пользоваться памятью без сдвигов массива при каждой операции

#deadlock_go_os
deadlock - состояние, когда несколько потоков\горутин навсегда ждут ресурсов друг от друга, и система без внешнего вмешательства не продолжит работу 
go:
- все горутины заблокированы на операциях с каналами, мьютексами или select без активных событий 
- рантайм проверяет состояние, если все горутины спят и никто не может ничего сделать, то кидает фатал
ос:
дедлок возникает при выполнении 4х условий (по Коффману):
- взаимное исключение(ресурс нельзя разделять)
- удержание и ожидание(процесс держит один ресурс и держит другой)
- отсутствие принудительного изъятия(ресурс нельзя отобрать)
- циклическое ожидание(есть цикл процессов, каждый ждет ресурс слудующего)
